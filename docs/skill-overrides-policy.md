# Skill Overrides Policy (Phase 62.2.5)

> **Status**: Active (2026-05-07)
> **Scope**: Claude Code `2.1.129` now supports the 3 modes `off` / `user-invocable-only` /
> `name-only` for the `skillOverrides` setting. This clarifies its relationship to Harness's
> skill governance and sets appropriate defaults for enterprise / individual use.

## In One Line

The `skillOverrides` 3 modes are a policy for **whether to let the model override skills**.
Harness **sets nothing by default** (= CC default behavior), but for enterprise governance it
recommends `name-only`.

## As an Analogy

It is a policy that decides "how much you let the chef (the model) tweak the recipe (the skill)."
- `off`: no tweaking (run only the default skill)
- `user-invocable-only`: allow only the tweaks the user names
- `name-only`: allow tweaking only what is specified by name; do not pull in other recipes on its own
- unset: CC default = some tweaking allowed

## Meaning of the 3 Modes

| mode | Meaning | Use |
|------|---------|-----|
| `off` | Fully disables skill activation via the model | Advanced enterprise environments, when you want to fully fix skill behavior |
| `user-invocable-only` | Allows only skills the user explicitly launches with `/<skill>`; forbids automatic model activation | Intermediate governance where you want to avoid "the model implicitly calling skills" |
| `name-only` | Allows activation only by matching the skill's name field (suppresses description-based auto-triggering) | When you want to prevent unexpected skill activation from fuzzy description matching |
| unset (default) | CC default behavior. Everything, including description-based auto-triggering, is enabled | Individual development, Harness default |

## Harness Default Policy

| Environment | Recommended mode | Reason |
|-------------|------------------|--------|
| Individual / solo development | unset (CC default) | Description-based auto-triggering is convenient |
| Team (small) | unset + skill manifest audit | Make the skill list visible with `scripts/generate-skill-manifest.sh` |
| Enterprise governance | **`name-only`** | Suppresses fuzzy description matching and allows only explicit skill activation |
| Education / training session | `user-invocable-only` | Forbids automatic model activation and has learners actively choose skills |

The template generated by `harness-init` **does not include a default** (= respects the CC default).
Enterprise users should set it explicitly in `.claude/settings.json` or `.claude/settings.local.json`.

## Relationship to the skill manifest

In Phase 59.1.2, `scripts/generate-skill-manifest.sh` was made to emit metadata such as `kind` /
`purpose` / `trigger` / `shape` / `role` / `base` / `pair` / `owner` in a machine-readable form.

In a `skillOverrides: name-only` environment, CC matches only the skill's **name**.
Description-based auto-triggering is disabled.
For that reason, skill names should be **semantically explicit** (verb + noun, such as `harness-work` / `harness-review`).

| Skill naming | Behavior in name-only mode |
|--------------|----------------------------|
| `harness-work` | Explicit launch is OK (`/harness-work`) |
| `breezing` (alias) | Explicit launch is OK (`/breezing`) |
| `harness-loop` | Explicit launch is OK (`/harness-loop`) |
| Abstract / generic word (e.g. `helper`) | Avoid, due to name-collision risk |

## Configuration Examples

### Enterprise governance (`.claude/settings.json`)

```json
{
  "skillOverrides": "name-only"
}
```

### Per-case disabling (specific environments only)

```json
{
  "skillOverrides": "off"
}
```

This lets you fully stop implicit skill activation in automated batch runs.

### Default (explicit setting not recommended)

There is no mode to explicitly specify `default`, so leave it unset to keep the CC default.

## Handling in tests / `harness-init`

- `tests/test-settings-baseline.sh` (creation considered in Phase 62.1.4) **allows but does not require**
  the presence of `skillOverrides` (so as not to hinder `default` in individual development)
- `harness-init` **does not include** `skillOverrides` in generated settings
- If enterprise governance is needed during porting / customization, refer to this doc
