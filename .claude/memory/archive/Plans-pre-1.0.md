The destination the whole task ledger aims at. Oldest first (foundation → top). The detailed contract is canonical in `spec.md`; this is a reference block.

- **L2 Tool-agnostic**: The same Harness (R01-R13 guardrails + plan/work/review/release) works "from any of" Claude / Codex / Cursor. One policy engine adjudicates 3 hosts via native hooks (routing, not duplication). Supports both directions equally — #1 harness drives (Lead spawns other tools as engines) / #2 used from the host (using harness "from" Codex/Cursor) (`spec.md` Execution Backend Contract / Host Adapter).

Mode 1 = fully autonomous orchestration (in v1 Lead=Claude is fixed; Codex/Cursor have no outbound spawn API). Mode 2 = human-present peer co-drive (live notice messaging). Full peer-Lead collaboration is deferred to a phased rollout (Phase 92 Purpose / `spec.md` Mode 1/Mode 2).

## Marker legend

Standard markers used in PM ↔ Impl operation:

| Marker | Meaning | Who sets it |
|---------|------|-----------|
| `pm:requested` / `pm:pending` | PM has filed the task and is requesting Impl | PM |
| `cc:todo` / `cc:TODO` | Impl task not yet started | Impl |
| `cc:wip` / `cc:WIP` | Impl (Claude Code) in progress | Impl |
| `cc:done` / `cc:done` | Impl finished the work, awaiting PM confirmation | Impl |
| `pm:approved` / `pm:confirmed` | PM completed final confirmation | PM |
| `cc:withdrawn` | Task withdrawn by Impl's judgment (superseded / absorbed into another task). breezing does not pick up cc:withdrawn | Impl |

**State transitions**: The canonical output on create/update is `pm:requested → cc:todo → cc:wip → cc:done → pm:approved`. The existing `pm:pending → cc:TODO → cc:WIP → cc:done → pm:confirmed` is also read-compatible. `cc:withdrawn` is a terminal state (never resumed).

**Backward compatibility**: the Cursor PM notations `cursor:requested` / `cursor:confirmed` are treated as synonyms of `pm:pending` / `pm:confirmed`.

## Phase 90: Orchestration Visibility Scorecard (session-scoped) [P1]

Purpose: Backend delegation is invisible at runtime, and the user cannot confirm "did it really use Codex / Cursor, or did it all fall back to Claude". Currently neither agent-trace nor telemetry records the backend dimension at all (`scripts/generate-agent-telemetry.js` aggregates the role dimension only), and companion scripts leave no delegation trace. This Phase wires the backend mix via "observe → record (always) → show (on-demand)". The key is to **separate recording and display**: delegation is always recorded in the session ledger, and rolls up idempotently into the lifetime accumulator at all-tasks completion (and, as insurance, at session end). Display is not emitted per task; a summary is emitted once to the terminal at all-tasks completion + HTML only when wanted. The cumulative total is "the star number to show people".

Finalized design (user decision reflected, updated from the previous session-only): (1) Recording = always ON. It has 2 scopes: the session ledger (that run's delegations) + the lifetime accumulator (`.claude/state/orchestration-totals.json`, per-backend cumulative per project). Roll up is **idempotent per session_id** (prevents double-counting across the 2 paths of completion/end), (2) Display = not emitted per task. **A terminal summary is auto-displayed exactly once at all-tasks completion** (running roll up first to reflect the session into lifetime). The HTML scorecard is on-demand only (does not open automatically). Both display session mix + lifetime totals, (3) Observation = companion is canonical (`codex-companion.sh` / `cursor-companion.sh` append directly to the ledger on delegation). The Claude side does not go through companion, so it is derived and merged from the existing `agent-trace.jsonl` (role=worker) at scorecard generation. The global (cross-all-project) cumulative is out of scope this time for cross-project privacy, opt-in in the future.

Spec delta: Updated `spec.md` `## Orchestration Visibility Contract` (separation of recording/display / session ledger + lifetime accumulator (project-scoped, session_id idempotent roll up) / no per-task auto-display / terminal summary once at all-tasks completion + HTML on-demand / companion-written ledger is non-sensitive fields only + status polling excluded via the `counts` flag / tri-state `used`/`available`/`not-configured` with not-configured silent / multi-backend utilization ratio + diversity / render-html redaction defense-in-depth / degrade to "no delegations observed" when missing).
