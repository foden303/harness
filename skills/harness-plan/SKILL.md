# Harness Plan

Harness's unified planning skill.
It consolidates the following three legacy skills:

- `planning` (plan-with-agent) — turning ideas into Plans.md
- `plans-management` — task state management and marker updates
- `sync-status` — checking alignment between Plans.md and the implementation

## Quick Reference

| User input | Subcommand | Behavior |
|------------|------------|----------|
| "create a plan" / `/harness-plan create` | `create` | Spec delta / skip reason → generate Plans.md tasks |
| "add a task" / `/harness-plan add` | `add` | Append a new task to Plans.md |
| "mark it done" / `/harness-plan update` | `update` | Change the task marker to cc:done |
| "where are we?" / `/harness-plan sync` | `sync` | Reconcile and sync the implementation with Plans.md |
| `/harness-sync` | `sync` | Progress check (equivalent to the standalone sync surface) |
| `/harness-plan create` | `create` | Create a plan across the spec.md / Plans.md dual sources of truth |
| `/harness-plan list` | `list` | List the named Plans in `plans/manifest.json` |
| `/harness-plan switch <name>` | `switch` | Save the active plan to `.claude/state/active-plan.json` |

## Literal companion commands (CC 2.1.108+)

- `/recap`: when returning after a long break, re-pull a summary before entering `sync`
- `/undo`: alias of `/rewind`. Use it directly when you want to immediately revert the last plan update

## Subcommand details

### Standard planning quality contract

See [references/planning-quality.md](${CLAUDE_SKILL_DIR}/references/planning-quality.md)

`harness-plan` is the planning surface that produces the co-required planning output for the spec.md product contract and the Plans.md task contract.
Precedence stays `spec.md > sub-spec > Plans.md`.
Plans.md is the task ledger and the root `spec.md` is the product contract; do not break that hierarchy.
Do not drop the given information into Plans.md as-is.
When creating a plan or adding a large task, check the latest information, existing specs, memory, and multi-perspective discussion via TeamAgent / subagents,
and convert only the elements that should be adopted into this product's task contract.
`/harness-plan create` returns a `Spec delta` or `Spec skip reason` together with generated `Plans.md` tasks.
The output must always include a `Spec delta` or `Spec skip reason`.
The `Spec delta` / `Spec skip reason` is generated by Harness; the consumer only approves or amends it.

**Non-trivial planning gate**:

Planning that is not a one-off, minor task is treated as requiring TeamAgent or subagents.
Here, non-trivial means a request that spans multiple tasks / files / sessions or affects product behavior / API / data model / permissions / billing / external integrations / distribution surfaces / security.
When the Task tool is available, run the independent Product / Architecture / Security / QA / Skeptic perspectives.
When it is not available, state `subagents-not-used` explicitly and evaluate the same perspectives separately on your own.

The output of non-trivial planning must always include the following validations.

- `team_validation_mode`: `not_required_lightweight` / `native` / `subagent` / `manual-pass` / `unavailable`
- Consistency across `spec.md` / sub-spec / `Plans.md`
- Reinvention-prevention check via harness-mem / harness-recall / repo memory
- Whether it stays aligned with the product's purpose
- Whether there are any issues with security, permissions, secrets, or the supply chain
- Whether a lint / formatter baseline exists. If a plan includes source code changes and none is set up, place a setup task before the implementation tasks
- Whether it is a workable plan — i.e. whether test / smoke / CI / review / release gates are captured in the task DoD

Lightweight tasks may use `team_validation_mode: not_required_lightweight`.
Non-trivial planning uses one of `native` / `subagent` / `manual-pass`.
Do not leave it as `unavailable` while marking it Required.
Product / Architecture / Security / QA / Skeptic are validation perspectives, not agent_type names.
Request them as perspectives from the available TeamAgent / Task subagents; do not require spawning arbitrary agents.
The Security gate does not require actually reading secrets.
If reading `.env` or secrets becomes necessary, stop it as a Risk Gate and confirm using existing permitted guards / evidence.

**When to apply**:

- Creating a new plan with `create`
- Using `add` for a task that affects product behavior / API / permissions / billing / external integrations / distribution surfaces
- The user hands over an external product, a competitor, a spec proposal, an improvement idea, or comparison material
- There is a risk of conflict with existing specs or past decisions

**When it can be handled lightly**:

- An `update` that only changes markers
- A `sync` that only reconciles status
- Typo, format, or README/CHANGELOG only
- A narrow change whose correct answer is fixed by an existing spec and tests

**Quality flow**:
1. Decompose the input, and state the evaluation targets, scoring axes, and uncertain facts
2. Fetch the latest information. For external facts, prefer WebSearch / official docs / primary sources, and cross-check key points across multiple sources
3. Check existing specs, the root `spec.md`, Plans.md, README, docs, CLAUDE.md, and related skills
4. Check available memory surfaces, project-scoped, such as harness-mem / harness-recall / `.claude/agent-memory/` / `.claude/state/`
5. For non-trivial planning, use TeamAgent / Task subagents to run independent reviews from different perspectives such as Product / Architecture / Security / QA / Skeptic
6. For plans that include source code changes, check the lint / formatter baseline, and place a setup task first if it is not set up
7. Produce a neutral scoring review and classify into Required / Recommended / Optional / Reject
8. Report the proposal, the rationale, and the outcome in `$easy` form
9. Convert only the adopted proposals into the root `spec.md` / Plans.md / test tasks

### Lane Taxonomy + Stage Gate

Fast / Gate / Release are treated as **Plans metadata, not new skills**.
Do not change the 5-column Plans.md template (Task / Content / DoD / Depends / Status).
Metadata such as lane / stage / TDD / unknown is embedded at **the start of the Content or DoD**.

**Lane Taxonomy** — attach one of the following to each task's Content or the start of its DoD:

| Tag | Purpose |
|-----|---------|
| `[lane:fast]` | low-risk local work (refactor / docs / typo) |
| `[lane:gate]` | changes touching spec / workflow / mirror / guardrail (most feature work) |
| `[lane:release]` | changes that ship a public artifact / version / tag / GitHub Release |

**Stage gate** — planning output (`create` / product-impacting `add`) is structured in the following 5 stages:

1. **Validation / research** — leave research evidence and state unknown data
2. **Finalize the implementation plan** — fix the lane tag + DoD
3. **Implementation (TDD)** — attach `[tdd:required]` or `[tdd:skip:<reason>]` to each task
4. **Review** — capture the review artifact (`harness-review` or a companion review) in the DoD
5. **PR closeout** — evidence pack → PR body (route to a separate `[lane:release]` gate if a release is needed)

Output the stage gate as a planning-section heading or as a preamble to a task group,
and generate the tasks corresponding to each stage as Plans.md rows.
Release closeout may be delegated to a `[lane:release]` task or to `harness-release`.

**Unknown data contract** — data the AI cannot see / failed searches / missing fixtures / unavailable APIs are
explicitly marked **`unknown`** under the `not_observed != absent` principle. Do not assert `absent`.

| Observed state | Output |
|----------------|--------|
| Confirmed not to exist | `absent` (evidence required) |
| Search failed / unread / API unavailable / fixture missing | `unknown` |
| Confirmed fact | Normal evidence description |

The output contract of `create` / `add` leaves an `unknown` list or
`unknown_data: [...]` at the research stage (stage 1). The DoD may include "do not write unconfirmed assumptions as absent."

### create — plan creation

See [references/create.md](${CLAUDE_SKILL_DIR}/references/create.md)

Interview for ideas and requirements, and generate an executable Plans.md.

**Flow**:
1. Check the conversation context (extract from the preceding discussion or interview anew)
2. Ask what to build (max 3 questions)
3. **Planning quality check** (latest information, existing specs, memory, multi-perspective TeamAgent / subagent review, scoring)
4. Technical research (WebSearch)
5. Extract the feature list
6. **spec.md / Plans.md dual-source check** (Spec delta or Spec skip reason + Plans.md tasks)
7. Priority matrix (Required / Recommended / Optional / Reject)
8. TDD adoption decision (test design)
9. Generate Plans.md (with `cc:TODO` markers)
10. **Generate the pre-approval section** (plan-time pre-approval)
11. Guide to the next action

### create — pre-approval section (plan-time pre-approval)

When finalizing a plan with `create`, after emitting the Plans.md tasks and before approval, always generate a **pre-approval section**.
The goal is not to permit anything via a permanent allowlist, but to confirm the likely stop / ask events per work scope just once, up front, at plan approval time.

Extraction targets:

- Each task's target files, related paths, and expected change scope
- Verification commands written in the DoD, PR closeout commands, and external API / CLI calls
- `secret-read path` (`.env*`, `secrets/**`, `*.pem`, `*.key`, `.ssh/**`, `.aws/**`, `credentials`, etc.)
- External sends (`git push`, `gh pr create`, `gh api`, `curl` / API call, release / publish / deploy)
- Destructive operations (`rm -rf`, migration destructive step, force push, production apply)

Fixed format:

```text
## Pre-approval
- Item: <specific operation for secret-read / external-send / destructive>
  Reason: <one line on why it is needed for the DoD or task execution>
  scope: Phase <phase> / Task <task>
```

Output rules:

- The one-line `Reason` must not include secret values. Keep it to paths / command names / target services.
- At plan approval, present all pre-approval items at once and obtain the user's approval / denial.
- Record the approval result in `.claude/state/plan-preapprovals.json` as `plan-preapproval.v1`. The schema is `templates/schemas/plan-preapproval.v1.json`.
- The record keeps `item + one-line reason + scope (phase/task)`; `operations` enumerates `secret-read` / `external-send` / `destructive`, `paths` / `commands` / `targets` enumerate the targets, and it includes `decision` and `approved_at`.
- Confirmation happens only once, at plan approval. During `harness-work` / `breezing` execution, do not raise `AskUserQuestion` merely because of an already-declared item.
- Unplanned secret-read / external-send / destructive operations not in the record still stop at the runtime floor / ask as before. Do not narrow the safety net.
- Approving a secret-read is not permission to display secret values. Treat it as input for declaring the minimal necessary path and applying it per-run to the project config `runtimefloor.secretAllow` at the start of work.

### spec.md / Plans.md dual-source check (default)

Treat Plans.md as the "what should be done" task contract, and the root `spec.md` as the "what is correct" product contract.
Co-required planning output means both outputs are mandatory; precedence stays `spec.md > sub-spec > Plans.md`.
When the implementation could drift, update the root `spec.md` before generating Plans.md.
`create` and product-impacting `add` read the root `spec.md` every time.

Preferred save locations:

1. root `spec.md`
2. Only when the consumer repo has no root `spec.md`, an existing project spec / architecture / product compass
3. Only when the consumer repo has no root `spec.md`, `docs/spec/00-project-spec.md`
4. In a repo with existing conventions, a spec path that follows those conventions

Conditions that require creation/update:

- A task that decides user-visible behavior, API, data model, permissions, billing, or external integrations
- A task with multiple implementation approaches where the choice changes product behavior
- A task with signs in past or current conversation that "the spec was ambiguous and the implementation drifted"
- A task where Plans.md has the work content, but the project's correctness conditions are not in a stable document

Conditions where it is unnecessary:

- Typo, format, dependency bump, or README/CHANGELOG only
- A narrow refactor with no behavior change
- A fix whose correct answer is sufficiently fixed by an existing spec and tests

Output contract:

- `Spec delta`: when updating the product contract, write the target spec path and the changes
- `Spec skip reason`: when not updating the product contract, write the reason
- The `Spec delta` / `Spec skip reason` is generated by Harness; the consumer only approves or amends it
- Even for docs-only / mechanical tasks, leave a `Spec skip reason` in the task context / sprint contract
- Do not assert missing search results, unavailable memory, or unread files as absent. `not_observed != absent`
- Do not make the user write a spec from scratch. The agent creates a minimal delta from the existing spec and the input, and only surfaces decision branches when ambiguous

Reference:

- `docs/plans/spec-ssot.md`

### Session startup guidance on create completion (required)

When `create` finishes, do not end with just an explanation. Provide, as a set, the **startup command for a new session** and
**the first instruction prompt to enter right after startup**.

The priority order is:

1. Only one task is incomplete, or starting just the first task is natural
   - Startup command: `claude`
   - First input: `/harness-work <task number>`
2. There are multiple loosely-dependent tasks and progressing them together is natural
   - Startup command: `claude`
   - First input: `/breezing all`
   - Alternative: `/harness-work all`
3. Long-running execution or re-entry is expected
   - Startup command: `ENABLE_PROMPT_CACHING_1H=1 claude`
   - First input: `/harness-loop all`
   - Alternative: `/breezing all`

Include at least these 3 lines:

- `Startup command for a new session:`
- `First input after startup:`
- `Best suited for:`

Example:

```text
Startup command for a new session: claude
First input after startup: /breezing all
Best suited for: Phase 1 has multiple tasks and progressing them together is natural
```

When recommending the long-running path, also include the Claude Code session startup command:

```text
Startup command for a new session: ENABLE_PROMPT_CACHING_1H=1 claude
First input after startup: /harness-loop all
Best suited for: long-running tasks with waits over 5 minutes or spanning resumes
```

Notes:

- `scripts/claude-longrun.sh` is a development helper script in this repository and is not distributed to the consumer environment after plugin install
- Therefore, for consumer-facing guidance, always prefer the single-line `ENABLE_PROMPT_CACHING_1H=1 claude` command
- If you want to use the equivalent wrapper only during repository development, `bash scripts/claude-longrun.sh` may be used on a local checkout

**CI mode** (`--ci`):
No interview. Use the existing Plans.md as-is and only perform task decomposition.

### add — add a task

Add a new task to Plans.md.
For product-impacting additions, follow the "spec.md / Plans.md dual-source check" above and also output a `Spec delta` or `Spec skip reason`.

```
/harness-plan add task name: detailed description [--phase phase number]
```

The task is added with a `cc:TODO` marker.

### update — marker update

Change a task's status marker.

```
/harness-plan update [task name|task number] [WIP|done|blocked]
```

Marker mapping:

| Command | Marker |
|---------|--------|
| `WIP` | `cc:WIP` |
| `done` | `cc:done` |
| `blocked` | `blocked` |
| `TODO` | `cc:TODO` |

### sync — progress sync

Reconcile the implementation status with Plans.md, detecting and updating diffs.

See [references/sync.md](${CLAUDE_SKILL_DIR}/references/sync.md)

**Flow**:
1. Get the current state of Plans.md
2. Detect the Plans.md format (v1: 3 columns / v2: 5 columns)
3. Get implementation status from git status / git log
4. Check the agent trace (`.claude/state/agent-trace.jsonl`)
5. Detect diffs between Plans.md and the implementation
6. Propose automatic fixes for stale markers
7. Present the next action

**Retrospective** (default ON):
If there is at least one `cc:done` task, automatically run a retrospective.
Analyze estimate accuracy, block-cause patterns, and scope changes, and record the learnings.
It can be explicitly skipped with `sync --no-retro`.

### team mode / issue bridge

Keep Plans.md as the source of truth, and use GitHub Issue integration only in the opt-in team mode.

- Solo development does not use the bridge
- Team mode creates one tracking issue and generates a per-task sub-issue payload under it as a dry-run
- `scripts/plans-issue-bridge.sh` never actually updates GitHub; it always returns a dry-run payload
- This bridge does not make changes to Plans.md

Reference:

- `docs/plans/team-mode.md`

### named Plans

When using multiple Plans.md files, make `plans/manifest.json` the source of truth and select by name.

```bash
scripts/plan-registry.sh list

scripts/plans-issue-bridge.sh --plan roadmap --format markdown

```

Operating rules:

- Use only one named plan per run
- For long-running / CI / issue bridge, do not rely on the active pointer; pass `--plan <name>`
- The manifest path is relative to the project root only. Absolute paths, `..`, and symlinks outside the repo are rejected

## Plans.md format conventions

### Format

```markdown
# [Project name] Plans.md

Created: YYYY-MM-DD

---

## Phase N: phase name

| Task | Content | DoD | Depends | Status |
|------|---------|-----|---------|--------|
| N.1  | description | tests pass | - | cc:TODO |
| N.2  | description | 0 lint errors | N.1 | cc:WIP |
| N.3  | description | migration runnable | N.1, N.2 | cc:done |
```

**DoD (Definition of Done)**: describe a verifiable completion condition in one line. "Looks good" and "works fine" are prohibited. Make it decidable as Yes/No.

**Depends**: dependencies between tasks. `-` (no dependency), a task number (`N.1`), comma-separated (`N.1, N.2`), or a phase dependency (`Phase N`).

### TDD tags

A Plans.md task can write a tag in its content or DoD to make the TDD decision explicit.

| Tag | Meaning | `tdd_required` inference |
|-----|---------|--------------------------|
| `[tdd:required]` | this task requires writing a failing test first | `true` |
| `[tdd:skip:<reason>]` | this task skips TDD with a reason | `false`, `skip_tdd_reason=<reason>` |

Do not leave `<reason>` empty.
Examples: `[tdd:skip:docs-only]`, `[tdd:skip:no-test-framework-detected]`.

When there is no tag, infer `tdd_required` in the following order.

1. Plans.md tag: `[tdd:required]` / `[tdd:skip:<reason>]`
2. files: if it includes source implementation such as `src/`, `app/`, `cmd/`, `lib/`, `pkg/`, `internal/`, `go/`, then required
3. TDD inference: for docs-only or when there is no test framework, attach a skip reason and treat as not required

### optional briefs / manifest

`harness-plan create` attaches a brief only when needed.

- The project spec SSOT is a document that fixes the project-wide correctness conditions; create it only when needed
- For tasks that include UI, a `design brief`
- For tasks that include an API, a `contract brief`
- A brief is a supporting artifact that briefly fixes "what to build"; it does not replace Plans.md or the spec SSOT
- The list of skill frontmatter can be turned into machine-readable JSON with `scripts/generate-skill-manifest.sh`

Reference:

- `docs/plans/briefs-manifest.md`
- `docs/plans/spec-ssot.md`

### Marker list

| Marker | Meaning |
|--------|---------|
| `pm:pending` | Requested by PM |
| `cc:TODO` | Not started |
| `cc:WIP` | In progress |
| `cc:done` | Worker work complete |
| `pm:confirmed` | PM review complete |
| `blocked` | Blocked (always state the reason) |

### Path after plan finalization (non-engineer plan overview)

Once appending tasks to Plans.md is complete, propose `harness-plan-brief` so a non-engineer requester can judge the plan.
This is a "plan overview" screen that summarizes understanding, options, risks, and acceptance criteria on a single HTML page,
readable without technical knowledge. Use it to reach agreement before starting implementation.
